# API

The full OpenAPI schema is served live at `http://localhost:8000/docs` (Swagger UI) and `http://localhost:8000/redoc`. This document covers conventions, lifecycle, and worked examples.

## Conventions

- All writes that involve LLM, embedding, or clustering work are **asynchronous**. They accept the request, persist a `jobs` row, enqueue an arq job, and return `202 Accepted` with a `Location: /jobs/{id}` header. Poll `GET /jobs/{id}` for completion.
- Reads are synchronous.
- Errors follow `application/problem+json` (RFC 7807) — fields are `type`, `title`, `status`, `detail`.
- Topic listings are always **nested under a parent context** — either a conversation or a clustering run. There is no orphan `GET /topics` listing; the only flat topic route is `GET /topics/{id}` for primary-key lookup.

## Lifecycle of a conversation

```
1. POST /conversations
   { "turns": [{"role": "user", "content": "I want a refund"}, ...] }
   → 202 + { conversation_id, job_id, mode: "created", batch_seq: 1, turn_count: 2 }
   → Location: /jobs/{job_id}

2. GET /jobs/{job_id}
   → 200 + { status: "completed", result: { chunks_created: 3, chunks_assigned: 2, ... } }

3. POST /conversations
   { "conversation_id": "...", "turns": [{"role": "user", "content": "also it took 3 weeks"}] }
   → 202 + { conversation_id, job_id, mode: "appended", batch_seq: 2, turn_count: 3 }

4. GET /conversations/{id}/topics
   → 200 + [ TopicWithEmergence[], sorted by emergence_score desc ]
```

## Endpoint reference

### POST /conversations

Ingest a new conversation or append to an existing one.

**Body:**

| Field | Type | Required | Notes |
|---|---|---|---|
| `conversation_id` | UUID | no | Omit for a new conversation. Provide to append; unknown id returns 404. |
| `turns` | `Turn[]` | yes | At least one turn. |
| `metadata` | object | no | Arbitrary JSONB (e.g., `{"source": "slack", "thread_ts": "..."}`). |

**Turn shape:**

| Field | Type | Required | Notes |
|---|---|---|---|
| `role` | string | yes | Free-text speaker label. |
| `content` | string | yes | Must be non-empty. |
| `timestamp` | ISO datetime | no | Connector-supplied; preserved as data. Not generated by the server. |

**Response:** `202 Accepted` + `ConversationIngestResponse`. `Location` header points to the job.

### GET /conversations

Paginated list of conversations.

Query params: `limit` (1-200, default 20), `offset` (default 0).

### GET /conversations/{id}

Metadata only. Raw turns live in `data/raw/{id}.json` — fetch them via the filesystem if needed.

### GET /conversations/{id}/topics

**The primary endpoint for the PM use case.** Returns the topics this conversation contributes to, with per-conversation emergence stats.

**Query params:**
- `sort=emergence` (default) — sorts by `emergence_score DESC, new_count DESC`
- `sort=mentions` — sorts by `mention_count_in_conversation DESC`

**Each topic in the response includes:**

| Field | Source |
|---|---|
| `id`, `label`, `sentiment`, `pm_insight`, `status` | `topics` table (global) |
| `mention_count` | global mention count across all conversations |
| `new_count` | chunks in latest batch_seq |
| `old_count` | chunks in prior batches |
| `mention_count_in_conversation` | new + old |
| `emergence_score` | per-conversation emergence ratio |
| `is_emerging` | `emergence_score ≥ EMERGENCE_RATIO_THRESHOLD`, with the fresh-conversation special case |
| `snippets_in_conversation` | 3 sample chunks from this conversation nearest the topic centroid |

### POST /clustering-runs

Trigger a full UMAP+HDBSCAN re-clustering pass.

**Response:** `202 Accepted` + `{clustering_run_id, job_id}`. Watch `GET /clustering-runs/{id}` or `GET /jobs/{id}` for completion.

This is also scheduled automatically every `CLUSTERING_CRON_HOURS` (default 24) by the arq cron.

### GET /clustering-runs/{id}/topics

Topics that participated in a specific clustering run, with snapshot mention counts captured at run completion.

**Query params:**
- `sort=mentions` (default) — `mention_count_at_run DESC`

Each item is a `TopicInRun` — global topic fields plus `mention_count_at_run` and `was_new`.

### GET /clustering-runs/latest/topics

Convenience alias for "the current state of topics globally." Resolves to the most recent **completed** clustering run.

### GET /topics/{id}

Flat primary-key lookup. Returns the topic plus 5 exemplar snippets from chunks across all conversations.

### GET /jobs/{id}

Generic background-job poll. Job lifecycle:

```
queued → running → completed | failed
```

`result` is populated on `completed`. `error` is populated on `failed`.

### GET /healthz

Liveness probe. `200 + {"status": "ok"}` requires a working DB connection.

## cURL examples

```bash
# Ingest
curl -X POST http://localhost:8000/conversations \
  -H 'Content-Type: application/json' \
  -d '{
    "turns": [
      {"role": "user", "content": "Where is my refund?", "timestamp": "2026-01-15T10:00:00Z"},
      {"role": "agent", "content": "Looking into it now.", "timestamp": "2026-01-15T10:01:00Z"}
    ],
    "metadata": {"source": "support_chat"}
  }'

# Append
curl -X POST http://localhost:8000/conversations \
  -H 'Content-Type: application/json' \
  -d '{
    "conversation_id": "<from-previous-response>",
    "turns": [
      {"role": "user", "content": "It has been 3 weeks now."}
    ]
  }'

# Cluster
curl -X POST http://localhost:8000/clustering-runs

# Read topics for the conversation
curl http://localhost:8000/conversations/<id>/topics

# Read global topics
curl http://localhost:8000/clustering-runs/latest/topics
```
